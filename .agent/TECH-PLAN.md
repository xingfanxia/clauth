### TECH-3: Daemon anti-wedge watchdog + standby supervision

**Why:** The single highest runtime hero-invariant risk. StateLock::acquire (lock.rs:87) is a blocking file.lock() with no deadline, and switch_profile runs the /usr/bin/security subprocess INSIDE that critical section — a first-run ACL prompt, relocked keychain, slow home volume, or a CLI/TUI wedged holding the flock freezes the single-threaded run loop forever: no status.json rewrites, no auto-switch, and launchd's KeepAlive{SuccessfulExit=false} never restarts a hung-but-alive process. Also the release-build backstop for the cfg(debug_assertions)-only rank system. Paired with #35: serve()'s try_lock+exit-0 on lock contention means a launchd instance that loses the race to a manually-run `clauth daemon` exits clean and is never restarted until next login → no daemon overnight.

**Scope:** Files: src/daemon/mod.rs (add an AtomicU64 heartbeat updated at the end of each tick; spawn a watchdog thread that std::process::abort() if no tick completed within ~60s — boot()'s relink + atomic writes make restart safe; change serve()'s single-instance guard at :85 from try_lock+Ok(()) to a blocking lock_file.lock() STANDBY so the launchd instance takes over the instant the manual instance exits, keeping the plist's 'clean exit must not restart' comment valid), src/keychain.rs (wrap the Command::output() calls in a wait-with-deadline / kill-on-expiry so a stuck `security` cannot pin the flock indefinitely). NOT TOUCH: do not change lock RANK ordering or the RankedMutex system; do not remove the flock serialization of writes; do not alter switch_profile's switch logic; abort() only — no unsafe.

**Acceptance:** New test (inject a fake clock/short deadline + a tick that stalls): watchdog reaches the abort path within the deadline — `cargo test daemon::watchdog` green (assert via a test seam / injected panic-hook, not a real abort in-process). Keychain-timeout test: an injected command runner that hangs is killed within the deadline — `cargo test keychain::timeout` green. `cargo clippy --all-targets` clean with unsafe_code=deny. Manual (documented in ledger): start `clauth daemon` by hand, `launchctl kickstart gui/$UID/com.clauth.daemon`, confirm the agent instance BLOCKS (does not exit); `kill` the manual one and confirm status.json generated_at resumes advancing within one tick.

### TECH-6: pending_switch queue correctness (+ last_error skip reason)

**Why:** Account-switch core, TDD-able on the TECH-5 seam. drain_pending_switch drains the whole HashSet then `continue`s past a busy or diverged target — the entry is CONSUMED not re-queued — so a socket user-switch that arrives during a fetch window (routine after a ccsbar Refresh marks every profile busy) gets one drain attempt and silently evaporates after the daemon already replied {ok:true}, violating the protocol's 'ok=accepted; poll status.json to see it land' guarantee and the repo's 'no silent failures' rule. Separately, because socket inserts without an emptiness guard while the scheduler queues its own target, a same-tick {user, auto} pair drains in arbitrary HashSet order → the user's explicit choice can be silently overridden. Auto-switch itself self-heals (scan_auto_switch is level-triggered), so the hero invariant survives — hence medium — but the manual switch core is incorrect.

**Scope:** Files: src/daemon/mod.rs (replace pending_switch HashSet with an ordered VecDeque<(String, Origin{User|Scheduler})>; RE-QUEUE a busy/diverged target with a small retry TTL instead of dropping; apply last-writer-wins with User outranking Scheduler within a drain; a User switch CLEARS queued Scheduler targets; introduce an additive status.json `last_error {at,message}` recording the last drain skip/failure reason), src/daemon/socket.rs (enqueue with Origin::User), src/usage/scheduler.rs (:1350 scan_auto_switch enqueue with Origin::Scheduler), src/daemon/status_json.rs (emit last_error; SCHEMA stays 1 — additive), tests/inline/daemon_mod.rs (update TECH-5 pins + add: busy target re-queued not dropped; same-tick User A + Scheduler B → final active A; user switch clears queued scheduler target). NOT TOUCH: do not change scan_auto_switch's level-triggered re-fire (that IS the auto-switch self-heal); do not alter the leaf-drain-then-config-lock discipline; do not bump SCHEMA_VERSION (last_error is additive, ccsbar decodeIfPresent).

**Acceptance:** `cargo test daemon_mod` includes and passes: `busy_target_requeued_not_dropped`, `user_switch_outranks_same_tick_auto`, `user_switch_clears_queued_scheduler`. `clauth status --json | jq -e '.last_error'` key exists (may be null). `cargo test` green; `cargo clippy --all-targets` clean. `rg -n 'HashSet' src/daemon/mod.rs src/usage/scheduler.rs` no longer shows pending_switch as a HashSet.

### TECH-7: Cross-process state RMW atomicity (lost-update)

**Why:** Genuine unaccepted cross-process lost-update on the state file, TDD-able on the TECH-5 seam. with_state_lock only serializes the write closure; every writer loads AppConfig OUTSIDE the flock (daemon at reload_if_changed, CLI at main.rs:183) then rewrites the whole profiles.toml from a stale in-memory snapshot via save_app_state. A `clauth login C` (or a switch sitting at the unbounded [Y/n] reconcile prompt) concurrent with a daemon auto-switch tick can rewrite profiles.toml WITHOUT C, and load_config never rescans the profile dir, so C is permanently orphaned. Compounded by the mtime self-adoption gap: last_state_mtime is read AFTER the save, so an external write landing in the save→read window is adopted as the daemon's own and never reloaded. Low probability + recoverable (re-login) → medium, but real silent data loss with a sound root-cause fix.

**Scope:** Files: src/lock.rs (make with_state_lock reload AppState from disk INSIDE the flock and apply the operation's delta before saving — pass a reload-and-reapply closure), src/actions.rs (:256 finish_switch owns only active_profile), src/fallback_config.rs (chain primitives own only fallback_chain/wrap_off), src/profile.rs (login/capture_into_profile owns only its profile entry; save_app_state merges rather than blind-overwrites), src/daemon/mod.rs (capture app_state_mtime() WHILE holding the flock so an external write is never mistaken for the daemon's own — closes the :354 self-adoption gap). NOT TOUCH: do NOT attempt to fix the TUI↔daemon chain-edit race here — that is the R3-accepted deferral (PROGRESS.md:85-91); scope strictly to the CLI-login and CLI-switch-prompt lost-update vectors; do NOT act on the refuted ccsbar fallback_add vector (in-process, cannot race itself).

**Acceptance:** New test: simulate an external write between the daemon's load and save (write a new profile to disk, then run a switch delta) and assert the merged save PRESERVES the external profile — `cargo test 'lost_update|rmw'` green. New test: app_state_mtime captured under the flock equals the value the reload compares against (no self-adoption window). `cargo test` green; `cargo clippy --all-targets` clean. ### concurrency (5 confirmed)

> Lock discipline inside the process is genuinely strong — typed compile-time ranks with sealed markers, documented nesting derivations, consistent drain-into-Vec-then-release patterns, snapshot-before-config in write_status, and a re-entrant cross-process state flock. ### daemon-protocol (7 confirmed)

> The daemon protocol's validation layer is genuinely good — dispatch is pure and unit-tested, unknown profiles/bad dirs/out-of-range thresholds are rejected at the socket, and all mutation funnels through the main loop. But the contract's middle layer is soft: 'ok means accepted' is not actually honored (accepted switches can be silently dropped with no re-queue and no error surface), there is no way to observe application (no sequence/ack — ccsbar guesses with a 1.2s sleep), errors are prose strings the Swift client discards entirely, and the transport itself has no line-size cap, no timeouts, and a single-threaded accept loop — one stalled connection freezes every command and can freeze ccsbar's main thread. On the schema side, the claimed 'lenient decode' protection is thin: only fallback_chain is lenient; every other field is strict, and a schema bump renders as the misleading 'daemon not running' state. None of these break the happy path today, but several sit one hop from the hero invariant (a daemon OOM from an unbounded read kills unattended auto-switch).

- **[medium/M] Socket-accepted switch is silently dropped when the target profile is busy — 'ok' means neither applied nor retained** — `src/daemon/mod.rs:328`
  dispatch() replies {"ok":true} after enqueueing into pending_switch (src/daemon/socket.rs:119-123), and the documented contract is 'ok = accepted; poll status.json to see it land' (socket.rs:17). But drain_pending_switch drains the whole set (mod.rs:322-326) and then does `if !is_idle(&self.activity, &name) { continue; }` (mod.rs:328-330) — the target has already been removed from the queue, so the switch is discarded, not deferred. Fetches mark profiles busy in the ActivityStore (src/usage/scheduler.rs:895,953,979 mark Queued/Fetching), so the window is real. The auto-switch path survives this only because scan_auto_switch re-fires while the threshold holds; a socket switch is one-shot. The divergence guard (mod.rs:335-345) and switch_profile failure (mod.rs:357) drop the command the same way, eprintln-only. This violates the protocol's one stated guarantee.
  **Fix:** Don't drop: re-insert busy targets into pending_switch so the next tick retries (with a small retry TTL to avoid a stuck target living forever), and record the last drain failure (e.g. a `last_error` field in status.json) so clients can surface 'switch skipped: profile busy / diverged credentials' instead of nothing.
